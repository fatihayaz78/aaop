Architecture Decision Records

All major architecture decisions are documented below. Each ADR captures the context, decision, and consequences.

ADRDecisionStatusRationale
ADR-001LangGraph for Agent OrchestrationACCEPTEDExplicit state machine control, deterministic 4-step loop, better than LCEL for multi-step agent workflows with branching and tool calls.
ADR-002FastAPI as Backend FrameworkACCEPTEDAsync-first, Pydantic v2 integration, OpenAPI auto-docs, excellent performance. Python 3.12 type hints fully supported.
ADR-003Next.js 14 for FrontendACCEPTEDApp Router, React Server Components, TypeScript-first, Tailwind + shadcn/ui component ecosystem. SSR/SSG flexibility.
ADR-004SQLite for Platform MetadataACCEPTEDZero-config, embedded, perfect for local dev. GCP migration path: swap to Spanner via adaptor pattern. Stores users, tenants, config.
ADR-005DuckDB for AnalyticsACCEPTEDColumnar OLAP engine, excellent for time-series aggregations. Embedded, no server needed. File: analytics.duckdb. GCP: BigQuery.
ADR-006ChromaDB for Vector StorageACCEPTEDSimple embedding storage and similarity search. Local persistent mode. Used for runbooks, knowledge base, prompt templates. GCP: Vertex AI Vector Search.
ADR-007Redis for Caching & Real-timeACCEPTEDIn-memory cache for active sessions, health status, rate limiting. Pub/Sub for WebSocket fan-out. GCP: Memorystore.
ADR-008asyncio.Queue Event BusACCEPTEDLightweight in-process event bus for inter-agent communication. 9 event types. No external dependency. GCP: could upgrade to Pub/Sub if needed.
ADR-009Anthropic Claude as LLM ProviderACCEPTEDThree tiers: Haiku (batch/cheap), Sonnet (default agent), Opus (P0/P1 critical). LLM Gateway handles routing, retry, caching, cost tracking.
ADR-010Adaptor Pattern for DB ClientsACCEPTEDAll DB access through shared/clients/ adaptors. Agent code never imports DB drivers directly. Enables GCP migration by swapping only the adaptor layer.
ADR-011Separate logs.duckdb for Raw LogsACCEPTEDRaw parsed CDN/origin logs in dedicated DuckDB file. Prevents log volume (millions of rows) from impacting analytics.duckdb query performance.

Agent 4-Step Loop

BaseAgent LangGraph State Machine

Every agent in AAOP extends BaseAgent from shared/agents/base_agent.py. The LangGraph state machine implements a consistent 4-step loop:

1. PLAN
Parse intent
Select tools
Determine strategy
→
2. ACT
Execute tools
Check risk level
Await approval if HIGH
→
3. OBSERVE
Aggregate results
Cross-reference data
Detect anomalies
→
4. RESPOND
Format output
Emit events
Store decision

Step Details

StepInputProcessingOutput
1. Plan User message + conversation history + system prompt LLM analyzes intent, selects relevant tools, creates execution plan. Multi-tool plans are parallelized where possible. Tool call list with parameters
2. Act Tool call list from Plan step Tools executed sequentially or in parallel. Each tool checks risk_level. HIGH risk tools pause for human approval. All tool calls logged with tenant_id. Tool results (success/error)
3. Observe Tool results + original context LLM synthesizes tool outputs, cross-references with platform state, identifies patterns or anomalies. May trigger additional tool calls (loop back to Act). Synthesized analysis
4. Respond Synthesized analysis Format response for the user. Emit relevant events to Event Bus. Store AgentDecision record in DuckDB for audit trail. User-facing response + side effects

ModelWhen UsedLatency Target
HaikuBatch operations, classification, embedding generation< 500ms

Gateway Features
Retry with exponential backoff (3 attempts), response caching (Redis, TTL: 5min for identical queries), cost tracking per tenant/model, rate limiting per tenant.

Event Schema (BaseEvent)

class BaseEvent(BaseModel):
    event_id: str = Field(default_factory=lambda: str(uuid4()))
    event_type: str              # One of the 9 event types
    tenant_id: str               # Required: multi-tenant isolation
    severity: SeverityLevel      # INFO | WARNING | ERROR | CRITICAL
    source_app: str              # Publisher app name
    payload: dict                # Event-specific data
    timestamp: datetime = Field(default_factory=datetime.utcnow)
    correlation_id: str | None   # For event chain tracking

GCP Migration Path

Key Principle: Agent Code Never Changes
The Adaptor Pattern (ADR-010) ensures that migrating to GCP only requires swapping the shared/clients/ layer. All agent code, tools, business logic, and API endpoints remain identical.

Migration Steps

  1. Create GCP project and enable required APIs (Spanner, BigQuery, Vertex AI, Memorystore, Secret Manager)
  2. Implement new adaptor classes that conform to the same interface as local clients
  3. Update shared/utils/settings.py to detect environment (local vs GCP) and load appropriate client
  4. Run data migration scripts to move existing data to GCP services
  5. Update CI/CD pipeline for GCP deployment (Cloud Run or GKE)
  6. Run full test suite against GCP backends to verify parity
  7. Gradual traffic migration with rollback capability

PII Scrubber

Located at shared/utils/pii_scrubber.py. Runs before any data is sent to the LLM Gateway. Detects and redacts:

  • Email addresses → [EMAIL_REDACTED]
  • Phone numbers → [PHONE_REDACTED]
  • IP addresses → [IP_REDACTED]
  • Credit card numbers → [CC_REDACTED]
  • Turkish national ID (TCKN) → [TCKN_REDACTED]

Authentication & Authorization

  • JWT-based authentication (PyJWT)
  • Access token (15min) + Refresh token (7d)
  • All API endpoints require valid JWT (except /health, /auth/login)
  • X-Tenant-ID header required for all data-access endpoints
  • Rate limiting via Redis (configurable per tenant)
Red Lines (9 Rules)
  1. No print() — use structlog exclusively
  2. No Python 3.14 — project is locked to 3.12
  3. No kaleido 1.x — must use kaleido==0.2.1
  4. No direct DB access in agents — always through shared/clients/
  5. No tool calls without tenant_id — multi-tenant isolation is mandatory
  6. No secrets in code — all secrets via .env (local) or Secret Manager (prod)
  7. No HIGH risk tools without approval — approval_required=True
  8. No LLM calls without PII scrubbing — pii_scrubber must run first
  9. No agent-to-agent direct RPC — communicate via Event Bus only
