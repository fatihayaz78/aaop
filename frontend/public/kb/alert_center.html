Overview

The Alert Center is the unified alert routing and management hub for AAOP. It receives events from all platform apps, deduplicates them, applies suppression rules, detects alert storms, and routes notifications to the appropriate channels based on severity.

Core Capabilities

  • Fingerprint-based deduplication with configurable TTL (default 900s)
  • Alert storm detection: >10 alerts in 5 minutes triggers suppression
  • Severity-based routing: P0 to PagerDuty, P1-P2 to Slack, P3 to email
  • Maintenance window suppression rules
  • Real-time WebSocket stream for NOC dashboards
  • Full CRUD for routing rules and notification channels

Routing Logic

5-Step Pipeline

Step 1

Dedup Check

SHA256 fingerprint lookup in Redis. If exists within 900s, alert is dropped.

→
Step 2

Suppression

Check maintenance windows and suppression rules in SQLite.

→
Step 3

Storm Detection

Count alerts per tenant in 5min window. If >10, suppress and send summary.

→
Step 4

Severity Routing

Route to channels based on severity level (P0-P3).

→
Step 5

DB Write

Write alert record to DuckDB alerts_sent table.

Fingerprint Algorithm

Dedup Fingerprint

fingerprint = SHA256(event_type + severity + tenant_id + affected_service)

The fingerprint is used as the Redis key suffix. If the same fingerprint appears within the 900-second window, the alert is considered a duplicate and silently dropped.

Redis key: alert:dedup:{tenant_id}:{fingerprint} with TTL=900s

Alert Storm Handling

Storm Suppression (HIGH Risk)

When more than 10 alerts from the same tenant arrive within 5 minutes:

  • Individual alert routing is paused
  • A single summary alert is generated (requires approval)
  • Summary includes count, severity distribution, and affected services
  • Storm tracker resets after suppression is acknowledged

Event Flow

The Alert Center is the universal receiver for all platform events. Every app publishes events to the event bus, and the Alert Center evaluates each one through the 5-step routing pipeline. It does not publish outbound events to other apps — it is the terminal notification layer.

Storage Architecture Note

Routing rules and channel configs live in SQLite (platform.db) for fast local reads. Alert history goes to DuckDB (shared_analytics) for cross-app analytics. Dedup state is in Redis for sub-millisecond lookups. In production, SQLite becomes Cloud Spanner and Redis becomes Memorystore.

Fingerprint Implementation

compute_fingerprint()

import hashlib

def compute_fingerprint(
    tenant_id: str,
    source_app: str,
    event_type: str,
    severity: str,
) -> str:
    """SHA256 fingerprint for dedup.
    Same combination within 900s = duplicate."""
    raw = f"{event_type}:{severity}:{tenant_id}:{source_app}"
    return hashlib.sha256(raw.encode()).hexdigest()[:16]

The fingerprint is truncated to 16 hex chars (64 bits) to keep Redis keys compact while maintaining very low collision probability for the expected alert volume.

Alert Lifecycle

Status Transitions

active ─────────> acknowledged ─────────> resolved
   │                    │
   │                    └──> escalated (if not resolved within SLA)
   │
   └──> suppressed (maintenance window or storm)
   │
   └──> deduplicated (silently dropped, not stored)
  • active — Alert routed to channel(s), awaiting response
  • acknowledged — Operator confirmed receipt, working on it
  • resolved — Issue fixed, alert closed
  • escalated — SLA breached, escalated to next tier
  • suppressed — Blocked by suppression rule or storm handler
  • deduplicated — Matched existing fingerprint, silently dropped

Testing Alerts

Use POST /alerts/test to verify channel connectivity without triggering a real alert. The test endpoint bypasses the dedup and storm detection pipeline and sends directly to the specified channel with a [TEST] prefix in the message.
