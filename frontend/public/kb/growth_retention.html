Retention Campaign Types

Win-Back

Targets churned users with special offers. Triggered 7 days after subscription cancellation.

At-Risk

Proactive offer for users with churn score ≥ 0.7 but still active. Personalized content recommendations.

Engagement Boost

For users with declining session frequency. Highlights new content matching their watch history.

HIGH Risk Tool: send_retention_campaign queues a campaign for batch delivery. Requires admin approval. Campaign targets are defined by user_id_hash segments, never raw PII.

Example Queries

# User asks: "What is the churn rate by region in the last 30 days?"

# Generated SQL:
SELECT
    region,
    COUNT(CASE WHEN churned = true THEN 1 END) * 100.0 / COUNT(*) as churn_rate_pct,
    COUNT(*) as total_subscribers
FROM subscriber_metrics
WHERE recorded_at >= CURRENT_DATE - INTERVAL '30 days'
GROUP BY region
ORDER BY churn_rate_pct DESC;

Data Model

DuckDB tables for growth analytics and subscriber metrics.

subscriber_metrics (DuckDB)

CREATE TABLE subscriber_metrics (
    id              VARCHAR PRIMARY KEY,
    tenant_id       VARCHAR NOT NULL,
    user_id_hash    VARCHAR NOT NULL,
    region          VARCHAR,
    subscription_plan VARCHAR,
    subscription_start DATE,
    churn_score     DOUBLE DEFAULT 0.0,
    churned         BOOLEAN DEFAULT FALSE,
    last_active     TIMESTAMP,
    total_sessions  INTEGER DEFAULT 0,
    avg_qoe_30d     DOUBLE,
    complaint_count INTEGER DEFAULT 0,
    recorded_at     TIMESTAMP NOT NULL
);

retention_campaigns (DuckDB)

CREATE TABLE retention_campaigns (
    id              VARCHAR PRIMARY KEY,
    tenant_id       VARCHAR NOT NULL,
    campaign_type   VARCHAR NOT NULL,  -- win_back|at_risk|engagement_boost
    target_segment  TEXT NOT NULL,
    status          VARCHAR DEFAULT 'pending',  -- pending|approved|sent|completed
    target_count    INTEGER,
    conversion_rate DOUBLE,
    created_at      TIMESTAMP NOT NULL,
    approved_at     TIMESTAMP,
    sent_at         TIMESTAMP
);

analytics_queries (DuckDB)

CREATE TABLE analytics_queries (
    id              VARCHAR PRIMARY KEY,
    tenant_id       VARCHAR NOT NULL,
    question        TEXT NOT NULL,
    generated_sql   TEXT NOT NULL,
    row_count       INTEGER,
    execution_ms    INTEGER,
    created_at      TIMESTAMP NOT NULL
);
PII Policy: All subscriber data uses user_id_hash only. The DataAnalystAgent's query validator blocks any SELECT that attempts to join with PII-containing tables outside the analytics schema.
