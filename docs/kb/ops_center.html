Overview

The Ops Center is the central nervous system for AAOP's 24/7 incident management. It combines the M01 Incident Copilot for real-time detection, classification, and response with the M06 RCA Engine for autonomous root cause analysis.

Primary agent for incident detection, classification, and initial response coordination.

Model Routing

SeverityModelReason
P0Claude OpusMaximum reasoning for critical outages
P1Claude OpusHigh-priority service degradation
P2Claude SonnetStandard analysis quality
P3Claude HaikuFast batch classification

RCAAgent(BaseAgent)

Specialized agent for root cause analysis. Always uses Claude Opus for deep reasoning. Auto-triggered for P0/P1 incidents only.

Auto-trigger Rules

  • P0 incident detected → RCA auto-starts immediately
  • P1 incident detected → RCA auto-starts after 60s if unresolved
  • P2/P3 → Manual trigger only via POST /ops/rca/trigger

4-Step Agent Loop

1. Perceive

Ingest event from bus, fetch context from DuckDB + Redis

→

2. Reason

LLM analysis with severity-based model routing

→

3. Act

Execute tools (risk-gated), write to DB, publish events

→

4. Reflect

Log decision, update MTTR, check if escalation needed

Tools marked HIGH always return {"status": "approval_required"} and require human confirmation before execution. This is enforced at the BaseAgent level.

Response: IncidentDetail (includes RCA result if available, correlation data, timeline)

WS /ws/ops/incidents Real-time incident stream ▶

WebSocket connection for real-time incident updates. Emits events:

  • incident.created — New incident detected
  • incident.updated — Status or severity changed
  • rca.started — RCA analysis initiated
  • rca.completed — RCA finished with results

ChromaDB — Incident Similarity Search

Collection: incidents

When a new incident arrives, the agent generates an embedding from the incident title, description, and affected services. It queries ChromaDB for the top-5 most similar historical incidents to provide context for RCA and response decisions.

FieldIndexedDescription
documentYes (vector)Incident title + description concatenated
metadata.incident_idFilterUnique incident identifier
metadata.tenant_idFilterTenant isolation
metadata.severityFilterP0/P1/P2/P3
metadata.root_causeNoRCA result (if available)
metadata.resolutionNoHow it was resolved

Embedding model: text-embedding-3-small (OpenAI) with 1536 dimensions. In production, this migrates to Vertex AI Vector Search.

Incident Status Flow

Status Transitions

Valid status transitions for incidents:

open ──────> investigating ──────> resolved
  │                │                    │
  │                │                    └──> reopened ──> investigating
  │                │
  └──> escalated ──┘
  │
  └──> auto_resolved (by system if metrics normalize within 300s)
  • open — Initial state when incident is created
  • investigating — Agent or human is actively working on it
  • escalated — Escalated to on-call engineer (requires approval)
  • resolved — Issue confirmed fixed
  • auto_resolved — System detected metrics returned to normal
  • reopened — Resolved incident has recurred

Correlation Rules

Signal CombinationInferenceAuto Action
CDN error spike + QoE buffering increaseCDN-origin issue affecting playbackCreate P1, trigger RCA
CDN error spike + no QoE impactEdge-only issue, users unaffectedCreate P2, monitor
QoE degradation + live event activeCapacity or CDN overload during eventCreate P1, notify capacity_cost
Multiple P2 incidents in 10minSystemic issue developingEscalate to P1, trigger RCA

Agent Decision Logging

Every agent action is logged to shared_analytics.agent_decisions with full traceability:

  • Which model was used and why (severity-based routing)
  • Which tools were invoked and their results
  • Processing time in milliseconds
  • Bilingual output (summary_tr + detail_en)
  • Correlation IDs linking to related incidents, RCAs, and alerts
